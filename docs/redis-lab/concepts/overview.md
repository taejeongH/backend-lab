# Redis 개요

## Redis?

- **RE**mote **DI**ctionary **S**erver
- key-value 형태로 데이터를 저장하는 인메모리 데이터 저장소
- value에 String, Hash, List, Set, Sorted Set, Stream 같은 다양한 자료구조 사용 가능

## Redis가 자주 쓰이는 곳

### 1. 캐싱

- 가장 흔한 용도
- DB에서 조회한 결과를 Redis에 잠시 저장해두고, 다음 요청에서는 Redis에서 바로 조회하는 방식
- 반복적인 DB 조회를 줄여 DB 부하를 낮추고 응답 속도를 개선할 수 있음

### 2. 세션 저장소

- 서버가 여러 대일 때 로그인 세션을 Redis에 저장하여 모든 서버가 공유할 수 있음
- 특정 서버에만 세션이 저장되는 문제를 줄일 수 있어 부하 분산 환경에서 유리함

### 3. Rate Limit

- 사용자나 IP별 요청 횟수를 Redis에 저장하고 TTL을 함께 설정하여 일정 시간 동안의 요청 수를 제한할 수 있음

```redis
INCR rate-limit:user:1
EXPIRE rate-limit:user:1 60
```

### 4. 중복 요청 방어

- 결제 요청, 주문 요청처럼 중복 실행되면 안 되는 작업에 대해 임시 key를 저장해 중복 처리를 막을 수 있음

```redis
SET payment:request:abc123 processing NX EX 300
```

### 5. Pub/Sub 기반 메시지 전달

- Redis는 Pub/Sub을 지원하므로 간단한 실시간 알림이나 이벤트 전달에 사용할 수 있음
- 다만 Redis Pub/Sub은 메시지를 영구 저장하지 않으며, 구독자가 메시지를 받지 못하면 유실될 수 있음
- 따라서 내구성이 중요한 메시지 처리에는 Kafka, RabbitMQ 같은 메시지 브로커가 더 적합할 수 있음

## Redis가 임시 저장소로 적합한 이유

### 1. 빠른 속도

- Redis는 주요 데이터를 메모리에서 다루기 때문에 낮은 지연 시간으로 응답하기 쉽다.
- 명령 실행 관점에서는 대부분 싱글 스레드로 동작하여 하나의 명령이 원자적으로 처리된다.
- 이 때문에 내부 자료구조 접근에서 락 경합이 적고 처리 흐름이 단순하다.
- 단, 여러 명령을 조합한 로직까지 자동으로 원자성이 보장되는 것은 아니므로 주의해야 한다.

### 2. TTL 지원

- Redis는 key마다 TTL을 설정할 수 있다.
- 시간이 지나면 자동으로 사라져야 하는 임시 데이터에 적합하다.

예를 들어 다음 데이터는 TTL과 잘 맞는다.

- 콘텐츠 상세 캐시
- 인증 코드
- 임시 토큰
- 중복 요청 방어 key
- rate limit key
- 세션성 데이터

### 3. 다양한 자료구조 지원

- 단순 문자열뿐 아니라 Hash, List, Set, Sorted Set 등을 사용할 수 있다.
- 따라서 캐시뿐 아니라 카운터, 랭킹, 큐, 집합 연산 등 다양한 용도로 활용할 수 있다.

## TTL이 캐시에서 중요한 이유

- TTL이 없다면 캐시 key가 계속 쌓여 메모리 사용량이 증가할 수 있다.
- 원본 데이터가 변경되었는데 캐시가 계속 남아 있으면 오래된 데이터를 반환할 수 있다.
- TTL을 설정하면 일정 시간이 지난 캐시가 자동으로 만료되어 메모리 낭비와 stale data 문제를 줄일 수 있다.
- Redis 메모리가 부족해지면 `maxmemory-policy` 설정에 따라 key가 제거되거나, 쓰기 명령에서 OOM 에러가 발생할 수 있다.

## Redis를 영구 DB처럼 쓰면 위험한 이유

- Redis는 메모리 기반 저장소이므로 많은 데이터를 장기간 보관하기에는 비용 부담이 크다.
- RDB/AOF 같은 persistence 기능을 통해 장애 후 복구 가능성을 높일 수는 있지만, Redis의 주 강점은 빠른 임시 데이터 접근이다.
- TTL, eviction policy, 장애 상황에 따라 key가 사라질 수 있으므로 원본 데이터 저장소로 사용할 때는 신중해야 한다.
- 주문, 결제, 게시글, 사용자 정보처럼 영구 보관과 정합성이 중요한 데이터는 RDBMS에 저장하고, Redis는 캐시나 임시 상태 저장소로 사용하는 것이 안전하다.

## 데이터 성격에 따라 TTL 필요 여부가 달라지는 이유

- 모든 Redis key에 TTL이 필요한 것은 아니다.
- 시간이 지나면 의미가 없어지는 데이터에는 TTL이 필요하다.
  - 인증 코드
  - 비밀번호 재설정 토큰
  - 중복 요청 방어 key
  - rate limit key
  - 콘텐츠 상세 캐시
- 반대로 장기 유지가 필요한 데이터라면 TTL 없이 관리할 수도 있다.
  - 서비스 설정 캐시
  - 명시적으로 갱신하는 랭킹 데이터
  - 수동으로 무효화하는 캐시
