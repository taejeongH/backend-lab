# 캐시

## 캐시 전략

### Cache Hit / Cache Miss
- Cache Hit : 캐시 스토어(Redis)에 데이터가 있을 경우(빠름)
- Cache Miss : 캐시 스토어(Redis)에 데이터가 없을 경우 Database에서 가져옴(느림)

### 캐시 읽기 전략(Cache Read Strategy)

#### Look Aside(Cache Aside) 패턴
```text
1. Redis Cache에 검색하는 데이터가 있는 지 확인
2. Redis Cache에 검색하는 데이터가 없을 경우 Database 확인
3. DB에서 조회한 데이터를 Redis Cache에 업데이트
```

- 일반적으로 사용되는 기본적인 캐시 전략
- Redis 장애가 발생하더라도 DB에 요청을 전달함으로써 캐시 장애로 인한 서비스 문제는 대비 가능
- Cache Stor와 Data Store(DB)간 데이터 정합성 유지 문제가 발생할 수 있으며, 초기 조회시 무조건 Data Store를 호출 해야 하므로 단건 호출 빈도가 높은 서비스에 적합하지 않음 
- 반복적으로 동일한 쿼리를 수행하는 서비스에 적합한 아키텍처

> #### Cache Warming
> - DB의 데이터를 미리 Cache에 저장하는 작업
> - 서비스 초기에 트래픽 급증시 대량의 Cache Miss가 발생하는 것을 방지 가능

#### Read Through 패턴
```text
1. Redis Cache에 검색하는 데이터가 있는 지 확인
2. Redis Cache에 검색하는 데이터가 없을 경우 Database를 조회하여 자체 업데이트
3. Cache에서 데이터를 가져옴
```
- Cache Aside 방식과 비슷하지만, Cache에 저장하는 주체가 Server가 아닌 Cache 자체
- 직접적인 Database 접근을 최소화하고 Read에 대한 소모되는 자원을 최소화 가능
- 캐시에 문제가 발생하였을 경우 즉시 서비스 전체 중단으로 빠질 수 있기 때문에 Redis와 같은 구성 요소를 Replication 또는 Cluster로 구성하여 가용성을 높여야 함

### 캐시 쓰기 전략(Write Cache Strategy)

#### Write Back(Write Behind) 패턴
```text
1. 모든 데이터를 Cache에 저장
2. 일정 시간마다 DB에 Query 전송
```
- 데이터를 저장할 때 Database가 아닌 먼저 캐시에 저장하고, 특정 시점마다 Database에 데이터를 저장하는 방식으로 캐시가 일종의 Queue 역할
- Cache에 데이터를 모았다가 한 번에 Database에 저장하기 때문에 쓰기 횟수 비용과 부하가 감소될 수 있으나, 데이터를 옮기기 전에 장애가 발생하면 데이터가 유실될 수 있다는 단점이 존재
- 데이터베이스에 장애가 발생하더라도 지속적인 서비스를 제공 가능

#### Write Through 패턴
```text
1. Database에 저장할 데이터가 있을 경우 Cache에 먼저 저장
2. 즉시 Cache에서 DB에 저장
```
- Cache와 Data Store에 동시에 반영하는 방식
- 항상 동기화가 되어 있어 항상 최신정보를 가지고 있다는 장점을 가짐
- 결국 데이터를 저장할 때 캐시 저장과 DB 저장 2단계의 과정을 거치기 때문에 상대적으로 느림

#### Write Around 패턴
```text
1. 모든 데이터는 DB에 저장
2. Cache Miss가 발생하는 경우에만 DB와 캐시에도 데이터를 저장
```
- 속도가 빠르지만 Cache Miss가 발생하기 전에 데이터베이스에 저장된 데이터가 수정되었을 때 사용자가 조회하는 Cache와 Database간의 데이터 불일치가 발생
- Database에 저장된 데이터가 수정, 삭제될 때마다 Cache 또한 삭제하거나 변경해야 하며, Cache의 TTL을 짧게 조정하는 식으로 대처

### 캐시 읽기 + 쓰기 전략 조합

#### Look Aside + Write Around
```text
조회 : 캐시에서 데이터를 먼저 조회하고, 없으면 데이터베이스에서 조회
저장 : 데이터베이스에 직접 저장하고, 캐시는 업데이트하지 않음
캐시 갱신 : 캐시는 다시 데이터가 조회될 때만 갱신
```
- 가장 일반적으로 쓰이는 조합

#### Read Through + Write Around
```text
조회 : 캐시에서 먼저 데이터를 찾고, 없으면 캐시가 데이터베이스에서 데이터를 조회하여 반환하고 캐시에도 저장
저장 : 데이터베이스에만 저장하고, 캐시는 업데이트 하지 않음
- 캐시 갱신 : 캐시된 데이터가 변경되지 않은 상태로 남아있다가, 다음에 해당 데이터를 조회할 때 갱신
```
- 항상 DB에 쓰고, 캐시에서 읽을 때 항상 DB에서 먼저 읽어오므로 데이터 정합성 이슈에 대한 완벽한 안전 장치를 구성 가능

#### Read Through + Write Through
```text
조회 : 캐시에서 데이터를 먼저 조회. 캐시 미스가 발생하면 캐시가 데이터베이스에서 데이터를 가져와 캐시에 저장하고 클라이언트에 반환
저장 : 클라이언트가 데이터를 저장할 때, 캐시와 데이터베이스 모두 동시에 저장되어 항상 최신 상태 유지
캐시 갱신 : 데이터가 변경될 때마다 캐시도 즉시 업데이트되므로, 캐시와 데이터베이스는 항상 일관성을 유지
```

## TTL

TTL은 캐시 데이터가 Redis에 유지되는 시간을 의미한다.  
TTL을 어떻게 설정하느냐에 따라 데이터 최신성과 캐시 효율 사이의 균형이 달라진다.

### TTL이 너무 짧은 경우

#### 장점

- stale cache가 오래 유지되지 않는다.
- 데이터 변경이 빠르게 반영될 가능성이 높아진다.
- 최신성이 중요한 데이터에 유리하다.

#### 단점

- Cache Miss가 자주 발생한다.
- DB 조회 횟수가 증가한다.
- 캐시를 사용하는 효과가 줄어든다.

### TTL이 너무 긴 경우

#### 장점

- Cache Hit 비율이 높아진다.
- DB 조회 횟수를 줄일 수 있다.
- 변경이 적고 조회가 많은 데이터에 유리하다.

#### 단점

- 데이터가 변경되어도 이전 캐시가 오래 유지될 수 있다.
- stale cache 지속 시간이 길어진다.
- 실시간성이 중요한 데이터에는 적합하지 않다.

### 정리

TTL은 짧을수록 데이터 최신성에는 유리하지만 캐시 효율은 떨어진다.  
반대로 TTL이 길수록 캐시 효율은 좋아지지만 stale cache가 오래 유지될 수 있다.

따라서 TTL은 모든 데이터에 동일하게 적용하기보다, 데이터의 변경 빈도와 최신성 요구사항에 따라 다르게 설정하는 것이 좋다.
