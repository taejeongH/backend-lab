# [redis-004] TTL Strategy

## 실험 질문

TTL은 stale cache 문제를 완전히 해결하는가, 아니면 영향을 줄이는 보완책인가?

## 배경

캐시 무효화가 누락되면 Redis에는 이전 콘텐츠 정보가 남을 수 있다.

TTL을 설정하면 stale cache가 영원히 남는 문제는 줄일 수 있지만, 불일치 자체를 막는 것은 아니다.

## 구현할 것

- Content 상세 캐시 TTL을 10초, 1분, 5분으로 바꿔가며 실험
- DB 값을 직접 수정한 뒤 TTL 전후 조회 결과 비교
- TTL 없는 캐시와 TTL 있는 캐시 비교

## 실험 1. TTL 만료 전후로 stale cache 지속 시간이 제한되는가

### 실험 내용

1. 콘텐츠 상세 캐시에 TTL 20초를 설정한다.
2. 콘텐츠를 조회해 Redis에 캐시를 저장한다.
3. DB 값을 직접 수정한다.
4. TTL 만료 전후의 조회 결과를 비교한다.

### 실험 결과

* TTL 만료 전까지는 Redis에 저장된 이전 데이터가 조회된다.
* TTL 만료 후에는 Cache Miss가 발생하고, DB의 최신 데이터가 다시 Redis에 저장된다.
* TTL은 stale cache를 제거하는 것이 아니라, 유지 시간을 제한하는 역할을 한다.

## 실험 2. TTL 길이에 따른 cache hit/miss 차이를 확인한다

### 실험 내용

1. TTL을 10초, 1분, 5분으로 바꿔가며 같은 조회 흐름을 반복한다.
2. 각 TTL 설정에서 cache hit/miss 로그를 비교한다.

### 실험 결과

* TTL이 짧으면 Cache Miss와 DB 조회가 자주 발생한다.
* TTL이 길면 Cache Hit 비율은 높아지지만, 오래된 데이터가 더 오래 유지될 수 있다.
* TTL은 데이터의 변경 빈도와 조회 빈도를 기준으로 정해야 한다.

## 깨뜨려본 상황. 캐시 무효화 없이 TTL만으로 정합성을 보장하려는 경우

### 상황

1. 수정/삭제 시 캐시 삭제 로직을 제거한다.
2. 콘텐츠 상세 캐시에 TTL만 설정한다.
3. DB 값을 변경한 뒤 TTL 만료 전까지 조회한다.

### 결과

* TTL이 만료되기 전까지는 이전 데이터가 조회된다.
* TTL만으로 즉시 정합성을 보장하려면 TTL을 매우 짧게 설정해야 한다.
* 이 경우 Cache Miss가 자주 발생해 캐시 효율이 크게 떨어진다.

## 이번 실험에서 알게 된 점

TTL은 데이터 정합성과 캐시 효율 사이의 트레이드오프를 조절하는 값이다.

TTL이 짧으면 stale cache 유지 시간은 줄어들지만 DB 조회가 증가한다. 반대로 TTL이 길면 DB 조회는 줄어들지만 오래된 데이터가 더 오래 유지될 수 있다.

따라서 TTL은 캐시 무효화를 대체하는 수단이 아니라, 캐시 무효화 실패나 누락 상황에서 stale cache의 지속 시간을 제한하는 보완책으로 보는 것이 적절하다.
